## 二、定版配置(Kernel 7,WM=32)

```
BM=64, BN=128, BK=8          // 注意:不再是方形 block tile
WM=32, WN=64                 // warp tile(★小 warp tile 是甜点)
TM=8,  TN=4                  // thread tile(沿用 K4/K5)
blockDim = 128 (= 4 warps)   // 一维 block
grid = (N/BN, M/BM)

派生量(全部 constexpr 符号推导,不硬编码):
WSUBM = 4*TM = 32            // warp 内 4 行线程 × TM
WSUBN = 8*TN = 32            // warp 内 8 列线程 × TN
WMITER = WM/WSUBM = 1        // ★ 关键:WM=32 让这个=1,acc 减半
WNITER = WN/WSUBN = 2

单线程账本:
acc  = WMITER*WNITER*TM*TN = 1*2*8*4 = 64    // ★ 对比 WM=64 时的 128
regA = WMITER*TM = 8
regB = WNITER*TN = 8
```

### 五层坐标真值表(warp tiling 命门,查表不靠记忆)

```
第一层:tid → warp / lane
  warpIdx = tid / 32
  laneIdx = tid % 32

第二层:warpIdx → warp 在 block tile 的位置(warp 列方向数 = BN/WN = 2)
  warpRow = warpIdx / 2
  warpCol = warpIdx % 2

第三层:laneIdx → lane 在 warp tile 的位置(warp 内列方向线程 = 8)
  laneRow = laneIdx / 8
  laneCol = laneIdx % 8
```

C 元素行坐标 = 五层偏移叠加:

```
c_row = blockIdx.y*BM + warpRow*WM + wmiter*WSUBM + laneRow*TM + tm
c_col = blockIdx.x*BN + warpCol*WN + wniter*WSUBN + laneCol*TN + tn
```

列坐标完全对称(row→col, M→N, WM→WN, TM→TN, warpRow→warpCol, laneRow→laneCol)。

**铁律**:取 regA 用 laneRow(`laneIdx/8`,商);取 regB 用 laneCol(`laneIdx%8`,余)。同一个 laneIdx 拆出行列两个身份,各管一边。除数永远是"列方向线程数"=8,行方向线程数(4)只决定取值范围,不参与运算。

## 三、循环结构(8 层 + 搬运)

```
acc[64] = {0};                       // ★ bk 循环之外
for (bk = 0; bk < K; bk += BK) {
    搬 As(转置存 As[BK][BM],float4 读 global + 拆 4 标量散写);
    搬 Bs(As[BK][BN],float4 直读直写);
    __syncthreads();
    for (k < BK) {
        for (wm<WMITER) for (i<TM step4): regA ← float4(As[k][row])  // 取数集中在前
        for (wn<WNITER) for (j<TN step4): regB ← float4(Bs[k][col])
        for (wm<WMITER) for (wn<WNITER) for (tm<TM) for (tn<TN):     // 计算
            acc[((wm*WNITER+wn)*TM+tm)*TN+tn] += regA[wm*TM+tm]*regB[wn*TN+tn]
    }
    __syncthreads();                 // ★ 尾部 sync,防下轮搬运覆盖未读完的 shared
}
写回 C;                              // ★ bk 循环之外,acc 累加完才写
```

- 计算循环顺序 **iter 外 / TM·TN 内**(`wm→wn→tm→tn`):让 regA/regB 在内两层稳定复用,寄存器局部性最好。
- WMITER=1 时第一层循环只跑一次自然退化,**保留循环不写死**,便于换配置/模板化。
- As 转置(`As[BK][BM]`)是 float4 取 regA 的前提:取数 `As[k][row]` 沿 BM 连续;代价是搬运写 As 变跨 stride 散写(float4 读 global + 拆 4 标量写)。同 K5 套路。

## 四、★★★ 核心认知(全部 ncu 实证,portfolio 高价值)

### 1. Warp Tiling 的本质:用寄存器换 shared 复用

"想复用 shared memory,寄存器数量必然增加,因为要保存被复用的部分。" 三级复用:block 级→shared、warp 级→访问协同 + 寄存器展开、thread 级→寄存器。**warp 级复用不靠线程互访寄存器(寄存器私有铁律不破),而靠硬件 broadcast 合并"读相同 shared 地址"的请求 + 扩大每线程的寄存器复用展开。**

### 2. ★ 甜点是小 warp tile,不是大的(本 kernel 最大教训)

| 配置                | acc  | Reg  | Occupancy | L1/TEX | GFLOPS                |
| ------------------- | ---- | ---- | --------- | ------ | --------------------- |
| WM=64,WN=64(大)     | 128  | 202  | 24.8%     | 98%    | 1866(惨败,输 K5)      |
| **WM=32,WN=64(小)** | 64   | 113  | 49.4%     | 90%    | **3628(赢 K5 1.22×)** |

大 warp tile 复用更狠,却因 acc=128 → 寄存器 202 → occupancy 24% 被压垮,净亏。小 warp tile 在"warp 级复用"与"寄存器预算"间平衡:既破 L1 墙(98→90%),又保住够用 occupancy(49%)。**最优点是平衡而非极端。** 这是"算术强度有寄存器预算天花板"的又一次实证。差一个数量级的性能就藏在 WM 这一个常量里。

### 3. ★ float4 在 warp tiling 上完全失效(对比 K5 的剧本反转)

|                 | K5 (thread tiling)         | K7 (warp tiling)                       |
| --------------- | -------------------------- | -------------------------------------- |
| float4 前后 Reg | 96→64(暗线红利)            | 202→202(纹丝不动)                      |
| 原因            | 省的是零散标量 load 临时量 | 瓶颈是 128 个 acc 累加器,float4 碰不到 |

**同一个优化手段(float4),在不同瓶颈结构下效果完全相反。优化是否生效,取决于它打的是不是当前的真瓶颈。** float4 只省得了"load 临时量"这类寄存器,省不了"acc 累加器"这类。补充 v3 第 10 条:寄存器红利也分种类。

### 4. ★ Warp Tiling 需要"寄存器富裕 + L1 真撞墙"同时成立

两张卡的互补困局(4090 数据来自关机前实验):

- **1650S**:L1 撞墙(98%)→ 有 warp tiling 用武之地;但寄存器小,大 warp tile 爆 occupancy。**靠小 warp tile(WM=32)在墙内找到平衡点,成功。**
- **4090(WM=64)**:寄存器够,但 **K5 的 L1 才 84.8%(没撞墙)→ warp tiling 没瓶颈可解**,occupancy 16.5%,输 K5 27%。
- **认知**:warp tiling 的价值取决于 baseline 是否已 L1 撞墙。siboehm 的大收益来自"大卡 + baseline 已撞墙"两条件同时满足。

### 5. ★ L1/TEX 墙首次被压下(warp tiling 灵魂判据)

97.9% → 90.2%。这是从专题三一路追的那堵墙第一次松动,Compute(SM) 因此从 64.6% 冲到 84.1%——kernel 从访存受限转向接近计算受限。**但 90% 仍是次瓶颈,没完全破,后续仍有空间。**

### 6. autotuning 的正确时机:机制验证成功之后

autotuning 只能在"已能赢的机制"里找最优点,不能把打不过的机制变成打得过的。WM=32 验证成功(机制成立)后,才轮到模板化扫 WM/WN/TM/TN 找真甜点。**这是 Kernel 6 autotuning 的正统续集,现在时机到了。**

## 五、踩坑记录

### ★ 致命乌龙:两个注册表只改了一个(v3 工程债的活案例)

WM 从 64 改到 32 后,`./verify 10` 用的是新 kernel(verify.cu 注册表改了),`./bench 10` 用的是旧 kernel(**benchmark.cu 注册表忘改**)。导致 bench 报 1866 + ncu 显示 Reg=202(旧 WM=64 特征),而源码明明是 WM=32。排查了半天"为什么改了源码寄存器不变",最后发现是 benchmark 注册表 stale。**教训:改配置后两个注册表必须同步;"源码改了但 ncu 数不变"的第一怀疑是注册表/编译没对齐,不是编译器玄学。** 这把 v3 的"两注册表重复维护"工程债优先级提到最高。

### 其他坑(均已闭环)

- regB 索引 `wn*TM+j` 写成 TM(应 TN),越界 + 错位。
- acc 声明 + 写回 C 一度被错放进 bk 循环内 → 每个 K-tile 重新清零/写半成品 + beta 反复累加。必须 acc 在 bk 前、写回在 bk 后。
- lane 拆分除数反复纠结:行用 `/8`、列用 `%8`,除数都是"列方向线程数"8,不是行方向 4。
- float4 搬运 idx 漏乘 4:应 `(tid+load*128)*4`,不是 `tid+load*128*4`。

### ★ verify 判据缺陷(已修,需同步)

原判据 `abs_err > eps || rel_err > eps`(用 `||`),当 ref 接近 0 时 rel_err 被放大误报 FAIL。4090 上 K5(成熟 kernel)和 K7 用相同错误数据双双 FAIL,反证是判据问题非 kernel bug。**改为 numpy allclose 风格混合容差**:

```cpp
float tol = abs_eps + rel_eps * fabs(ref);   // abs_eps=1e-3, rel_eps=1e-2
if (fabs(mine-ref) > tol) bad_count++;
```

**两台机器的 verify.cu 都要同步此判据,且改后需做鉴别力测试(故意改错一处坐标必须 FAIL)。**
