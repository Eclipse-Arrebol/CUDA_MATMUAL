# CUDA Matmul 项目 —— 交接文档 v5 增补(Kernel 8: Bank Conflicts / 搬运重构)

> 本文件是 v4 的增补,记录对 Kernel 7(Warp Tiling 定版,本机 id 历经 10→11)做 bank conflict 优化的完整历程。
> 与 v3 / v4 配合使用。核心结论:**bank conflict 成功清零(16M→0),global 合并改善(16→8 sec/req),但 Elapsed Cycles 净平——bank conflict 是"ncu 显眼但非 binding"的瓶颈。这是本轮最高价值的认知。**

---

## 〇、一句话总结

针对 Kernel 7 写 As 的转置散写 bank conflict(op_st=16M),做了两步重构:**搬运映射改造(lane 沿 K 铺)+ p=4 padding**。结果 bank conflict 彻底清零、global sec/req 减半,但 **cycle 一个没省(净平)**。终审用 ncu Warp Stall 实证:这张卡上 kernel 已接近 compute-bound(math_pipe_throttle 高),shared 等待(short_scoreboard)垫底——证明 bank conflict 本就不在关键路径上。下一刀:double buffering(收官),收益预期有限,主做 latency hiding 实战理解。

## 二、做了什么(两步重构,严格一次一变量)

### 病灶定位(全 ncu 实证)
- 读 As(`As[k][row]`,内层 regA/regB):**op_ld=0,干净**。v4 那条"内层 regA/regB 读怀疑有 conflict"是**误报**。读 As 靠 broadcast:4 个 laneRow 落 4 不同 bank,每 row 被 8 lane 同址共享 → broadcast 受益,零 conflict。
- 写 As(`As[as_c][as_r]`,转置散写):**op_st=16M,真凶**。原映射 `idx=tid*4` 让 as_c∈{0,4} 两组,bank 错位重叠 → 2-way conflict。
- **同时**读 A 的 global 合并也烂:**16 sec/req**(原映射 lane 沿 M 铺,相邻 lane global 地址跨 K=4096 floats)。→ "两侧都散"的最差态。

### 第一刀:搬运映射改造(救 global 合并)
唯一关键改动:**`idx = tid*4` → `idx = tid`(去掉 *4),搬运循环 `load<1` → `load<4`,float4 读 A → 标量读**。
```cuda
for(int load=0; load<4; load++){
    int idx = tid + load*128;              // 无 *4 → 相邻 lane idx 跳 1
    int as_r = idx / BK, as_c = idx % BK;  // as_c=余=K方向快变(相邻lane连续0..7)
    int a_row = blockIdx.y * BM + as_r;
    int a_col = bk + as_c;
    As[as_c][as_r] = A[a_row*K + a_col];   // 标量读、标量写
}
```
- 效果:global sec/req 16→8(相邻 8 lane 沿 K 连续读)。
- 代价:写 As stride 变 BM=64,`64%32=0` → op_st 暴涨 16M→117M(中间态,下一刀治)。
- 失去读 A 的 float4 不亏:原版 16 sec/req 本就没享受到 float4 合并红利。

### 第二刀:p=4 padding(消 As 写 conflict)
```cuda
__shared__ float As[BK][BM+4];   // = As[8][68]
```
- op_st:117M → 0(彻底清零)。
- 注意:`As[8][68]` stride=68,`68*4=272`,`272%16=0` → 16字节对齐恢复,读 As 可保留 float4(但 v4 实测 float4 在读侧无收益,保持标量也行)。

## 三、★★★ 核心认知(本轮最高价值,portfolio 必留)

### 1. ★★★ ncu 的 throughput% ≠ binding bottleneck(本轮第一教训)
L1/TEX ~90% 看着像墙,但消除背后的 bank conflict(16M→0)对 Elapsed Cycles **零影响(净平)**。bank conflict 是"**ncu 上显眼、但不在关键路径**"的瓶颈——它占满了 L1 流水线的某个计数器,却没卡在决定总时间的依赖链上。
**判断一个瓶颈值不值得打,看的是它消除后 Elapsed Cycles 降不降,不是它在 ncu 上的 throughput% 有多高。** 很多人盯着 90% L1 一通优化,拿到 0 加速还困惑——区分"指标高"和"该优化"是编译器后端工程师的核心判断力。
> 补 v4 "优化是否生效取决于打的是不是真瓶颈":这次打的是**真实存在(ncu 实证 16M)但非 binding** 的瓶颈。工程上正确(代码更干净、可移植到真撞墙的卡有用),但这张卡这个尺寸下不加速。**负结果,价值高。**

### 2. ★★ padding 量由"快变维组数 × 慢变维宽度 = 32"反推,不是固定 +1
经典教材"bank conflict 就加 +1"是**特例**。一般规律:padding=p 时 `bank = (as_c*(BM+p) + as_r) % 32`,因 `BM%32=0` 化简为 `(as_c*p + as_r)%32`。要让同一拍 32 lane(as_c=0..7 共 8 组,各带 as_r=0..3 宽度 4)落 32 不同 bank,需 `(BM+p)%32 = 慢变维宽度 = 4` → **p=4**。
- p=1:`(as_c+as_r)%32`,as_c↓1、as_r↑1 时 bank 不变 → **斜条纹残留**(实测 117M→50M,只削一半)。
- p=4:`as_c*4+as_r` 对 (as_c=0..7, as_r=0..3) 是完美一一映射 → 归零。
**先给 conflict 归类(连续 stride 撞 bank 型 vs 离散值组错位型),再选 padding 量。**

### 3. ★★ padding 分两种命运:连续 stride 型有效,离散组错位型无效
- 原映射的 conflict:as_c∈{0,4} **两组离散值在 bank 错位重叠**。padding 改 `as_c*stride` 系数,改不了"as_c 只有两个值"的离散结构 → **+1 无效**(实测纹丝不动 16M)。
- 新映射的 conflict:as_c=0..7 连续,stride=64 撞 bank。**连续 stride 型** → padding 改 stride 打散 → **有效**。
**同一招(padding)对不同 conflict 类型命运相反。必须先用逐 lane 真值表归类,不能想当然开刀。**(本轮踩过:第一次没归类直接 +1,白做。)

### 4. ★ float4 + 奇数 padding 的对齐陷阱
padding 到奇数(BM+1=65)→ `As[k][row]` 地址 `k*65*4`,`260%16=4≠0` → float4 reinterpret_cast **未对齐**(WDDM 下 sanitizer 抓不到,埋雷)。
- 解法 A:读侧退标量(reg 红利在读侧本就为零,不亏)。
- 解法 B:padding 到 4 的倍数(p=4 → 68*4=272, %16=0)恢复对齐。**p=4 顺便解决了对齐,双赢。**

### 5. ★ global 流量主体是固有复用,coalescing 收益有限(算术,非猜测)
A 的一个 block 行块 `BM×K` 被 `N/BN = 32` 个 block 重复读 → **A 总读取量 = 32× A_size**。这是 shared tiling 的固有复用结构,coalescing/对齐**改不了这个 32×**。sec/req 16→8 只省冗余传输的边角,DRAM 主体降不下来(实测 66%→63%)。
**判断 coalescing 该不该花力气前,先算"数据被重读几遍"——固有复用流量是 coalescing 的天花板。**

### 6. Warp Stall 真凶榜(--set full / metrics,本轮终审)
```
not_selected       1.48   ← 最大,但是"健康 stall"(warp 就绪只是没排上,忙的标志)
math_pipe_throttle 1.43   ← 第二,FFMA 流水线饱和 → 接近 compute-bound(吻合 Compute 84%)
long_scoreboard    1.17   ← 第三,等 global/DRAM(double buffering 的靶子,中等)
wait               1.09   ← 算术依赖链 RAW
mio_throttle       0.80   ← LSU/shared 发射口
short_scoreboard   0.33   ← 最小,shared 等待 → 实证 bank conflict 已非 stall 来源
```
- `Issued Warp/Scheduler = 0.49`、`No Eligible = 51%`、`Eligible Warps = 1.22 / 3.95 active`:典型 **latency-bound + occupancy(49%)不足以填 stall 空窗**。
- **诊断**:1650S 上这个 kernel 已接近 compute-bound,延迟 stall 分散(long/wait/mio 各占一点),**没有单刀能解的大瓶颈**。short_scoreboard 垫底是今天工作"净平"的最干净佐证。

## 五、下一步 / 待办

### ★ 立即:Kernel 9 — Double Buffering(收官刀,latency hiding 实战)
**预期收益有限**(long_scoreboard 仅排第三 1.17,kernel 已近 compute-bound),主做专题五 latency hiding 的实战理解,不奔大加速。带着"可能又净平边缘"的预期去做。
- 双 buffer:`As[2][BK][BM+4]` + `Bs[2][BK][BN]`,搬 i+1 块同时算 i 块,用计算盖住搬运的 global 延迟。
- **动手前三个必答**:
  1. shared 翻倍后字节数 = ?(当前 + 翻倍,对照 1650S 每 SM 64KB 上限)→ **会不会反超成 occupancy 新限制项?**(当前限制项是寄存器,翻倍后要重新确认)
  2. 循环结构:"先搬第0块 → 循环{搬 i+1(不立刻 sync)→ 算 i → sync}→ 算最后一块"。哪个 sync 能省、哪个不能?
  3. 预测 cycle 能压多少?还是净平边缘?
- 验证三连:verify(布局大改,索引最易错)→ op_st/op_ld 仍为 0 → `--set full` 看 long_scoreboard 1.17 降多少 + Elapsed Cycles 对账。

### 收官后:转场决策
1650S 上已接近"每个瓶颈显眼但非 binding / 接近算力天花板(~3 TFLOPS,3628 GFLOPS 已是峰值相当部分)"。真正空间在:
- **4090**:L1 没撞墙(84.8%)、寄存器富裕、occupancy/算力天花板完全不同。v4 还欠一个 **4090 上 WM=32** 的数据点(关机前只测了 WM=64)。把本轮重构也同步到 4090,验证"不撞墙的卡上 bank conflict 优化是否同样净平甚至略亏"——闭环"throughput% vs binding"认知。
- **Tensor Core(WMMA/MMA)**:对"编译器后端"目标更对口(MLIR/TVM/TVM 后端大量在调度 MMA)。siboehm 路线(到 K10 double buffering)只是入门到中级,不是天花板。
