### Kernel 4(2D Block Tiling)定版配置 —— **TM=8, TN=4**

- `BM=64, BN=64, BK=8, TM=8, TN=4`
- blockDim = **(16,8) = 128 线程**;grid = CEIL_DIV(N,64)×CEIL_DIV(M,64)
- 一个 thread 算 C 里一个 **TM×TN = 8(行)×4(列)** 的小矩形
- **核心机制(2D vs 1D 的灵魂)**:内层先把 `regA[TM]`、`regB[TN]` 预取进寄存器,再做 TM×TN 次乘加,全程不碰 shared → **A 和 B 同时复用**(1D 只有 B 复用)
- 访存比:`(TM+TN):(TM*TN) = 12:32 = 1:2.67`(1D 是 ~1.1:1)

#### 三套坐标真值表(易错命门,务必查表不靠记忆)

| 量                  | 表达式(blockDim=16×8)                          |
| ------------------- | ---------------------------------------------- |
| 列方向 thread 数    | BN/TN = 16                                     |
| 行方向 thread 数    | BM/TM = 8                                      |
| 列序号              | `tid % 16`                                     |
| 行序号              | `tid / 16`                                     |
| 行起点(取 A / 写 C) | `(tid/16)*TM`                                  |
| 列起点(取 B / 写 C) | `(tid%16)*TN`                                  |
| acc 展平            | `acc[i*TM+j]`,i 管列(范围 TN)、j 管行(范围 TM) |

- **铁律**:取 A 的行 = 写 C 的行;取 B 的列 = 写 C 的列。三套坐标必须用同一套。
- **搬运**:128 线程搬 As/Bs 各 512 个元素 → 每线程搬 4 个,`idx = tid + load*128`,`load<4`,左边 shared 下标用 idx 算出的 as_r/as_c(不是 tid),右边 global 地址要带 `blockIdx.y*BM` / `bk` / `blockIdx.x*BN` 偏移 + 边界判断补 0。

#### Kernel 4 的 ncu 画像(4096³)

- L1/TEX **98.34%**(↑,从 1D 的 81%)—— **新瓶颈 = SRAM 子系统饱和**
- Compute (SM) 53%(↑,从 1D 的 48%)
- DRAM 44%(↑,从 1D 的 27%)
- Registers/Thread = **96**;Achieved Occupancy = **62%**(↑,从 1D 的 50%)
- **关键洞察**:occupancy 不降反升,因为 block 从 512→128 线程,单 block 寄存器总账更轻(128×96=12288),SM 能塞 5 个 block(Block Limit Registers=5)。**"tile 与线程排布解耦" 的深层回报。**
- 加速 1.8× 来源是双重的:访存比改善 + occupancy 意外提升。

## 四、重要反例:8×8 过界 spilling(博客高光,务必保留)——**注:此为无 float4 时的反例;float4 后已不 spill,见四点六**

试过把 TN 4→8(blockDim 改 8×8=64 线程,搬运改每线程搬 8 个、`load<8` 步长 64)。**前提:无 float4 版本。**

| 指标             | 8×4(赢)  | 8×8(输)                   |
| ---------------- | -------- | ------------------------- |
| GFLOPS           | **2308** | **1358**(↓,退回 1D 水平)  |
| Registers/Thread | 96       | **128(触顶台阶)**         |
| Compute (SM)     | 53%      | **31.65%(腰斩)**          |
| DRAM             | 44%      | 27.94%                    |
| Occupancy        | 62%      | 49%                       |
| verify           | PASS     | PASS(逻辑正确,纯性能问题) |

**结论:8×8 在 1650S 上触发 register spilling。**

- acc 升到 64 个 + regA/regB 16 + 索引开销 + O3 展开临时量 → 逻辑需求 150+,编译器压在 128,装不下的 spill 到 local memory(= DRAM)。
- 内层 64 次 `acc[...] +=` 从寄存器累加变成 local memory 往返 → Compute stall → duration 62ms→111ms。
- **核心认知**:算术强度有**寄存器预算天花板**,且天花板是**单卡特性**。siboehm 3090 能吃 8×8,1650S 不行。这正是 Kernel 6(autotuning)存在的理由——甜点是扫出来的,不是推出来的。

```
ncu --metrics l1tex__t_bytes_pipe_lsu_mem_local_op_ld.sum,l1tex__t_bytes_pipe_lsu_mem_local_op_st.sum ./bench <id> 4096 4096 4096
```

- 预期:旧 8×8(无 float4)的 local load/store 字节数 >> 0;float4 后的 8×8(id=8)应接近 0(已不 spill,见 Kernel 6)。非零即 spilling 实锤。

## 四点五、Kernel 5 —— float4 向量化访存(siboehm Kernel 6)定版 ~2970

### 核心机制

- 一条 `LDS.128`/`LDG.128`/`STS.128` 搬 4 个 float,把 shared/global 访问**指令数砍到 1/4**(数据量不变),给 LSU 卸压。
- `float4` 内建类型(16B),`reinterpret_cast<float4*>(&ptr)[0]` 把 float 地址当 float4 读,`.x/.y/.z/.w` 取 4 分量。类比 binary loader 里把 bytes 强转 struct 指针。

### 两个硬约束的落地

1. **16B 对齐**:float4 起址的 float 下标必须是 4 的倍数。`col_in_tile=(tid%col_threads)*TN`、`row_in_tile=(tid/col_threads)*TM`,TN=4/TM=8 均满足。
2. **As 转置存储**:这是核心难点。
   - 原 regA 沿行取 `As[row+i][k]`,跨行 stride=BK 不连续 → float4 取不了。
   - 解法:As 声明改 `__shared__ float As[BK][BM]`(转置),取数变 `As[k][row+i]` 沿 BM 连续。
   - **代价转移**:搬运写 As 从顺序写变跨 stride 散写;且因 global A 沿 K 连续、As 沿 M 连续**两头不可兼得**,搬运时 **float4 读 global(保住 LDG.128)+ 拆成 4 标量散写 As**(写 shared 带宽富余,认了)。省在热路径(内层 BK 次取数),亏在冷路径(每 K-tile 搬运一次),净赚。
   - regB 侧天然连续(`Bs[k][col+i]` 沿 BN),无需转置,直接 float4。

### Kernel 5 的 ncu 画像(4096³)与**反直觉发现**

| 指标                   | Kernel 4(2D, 无float4) | Kernel 5(float4)                 |
| ---------------------- | ---------------------- | -------------------------------- |
| L1/TEX                 | 98.34%                 | **97.91%(几乎没降!)**            |
| Compute(SM)            | 53%                    | 64.65%                           |
| DRAM                   | 44%                    | 56.87%                           |
| **Registers/Thread**   | 96                     | **64(↓↓)**                       |
| **Achieved Occupancy** | 62%                    | **98.77%(↑↑ 拉满)**              |
| ncu 诊断               | latency issues(<60%)   | **Compute/Memory well-balanced** |

- **★ 核心认知(博客高光)**:float4 的初衷是"砍 L1/TEX 指令",但 **L1/TEX 纹丝不动**。真正的提速来自暗线——**float4 让编译器寄存器分配更紧凑(96→64)→ Block Limit Registers 5→8 → occupancy 62%→98.77% → 满 warp 掩盖延迟,latency 警告消失。** 优化生效了,但生效的机制和预想完全不同。
- **★ 分阶段验证的关键**:只 float4 内层取数时性能纹丝不动(~2360);把搬运也 float4 化后才跳到 ~2970。证明 **L1/TEX 的大头是搬运的窄访存,不是内层取数**——瓶颈定位错方向则优化白做。

## 四点六、Kernel 6 —— Autotuning(siboehm Kernel 7)+ 8×8 spill 悬案闭环

### 工程实现

- **模板化**:`autotuning_kernel<BM,BN,BK,TM,TN>` + `launch_at<...>`,全部进 `include/06_autotuning.cuh`。

- **参数化要点(模板化最易栽处)**:

  - `threads=(BM/TM)*(BN/TN)`、`As_loads=(BM*BK)/(threads*4)`、`Bs_loads` 同理、`col_threads=BN/TN`。
  - **硬编码 `16`→`BN/TN`、`128`→`threads`**。三套坐标拆 tid 全用 `col_threads`。**血泪坑:`tid/(BN/TN)` 不能代数约分成 `tid*TN/BN`,整数除法不等价;`col_in_tile` 曾误写 `(tid*TN%BN)*TN` 差 4 倍。**
  - 搬运 `idx=(tid+load*threads)*4`,loads=1 时退化为 `tid*4`。

- **static_assert 守门**(编译期挡非法配置,不靠运行时崩):

  ```
  static_assert(BK % 4 == 0, ...);                    // As float4 不跨 BK
  static_assert((BM*BK) % (threads*4) == 0, ...);     // As 每线程整数个 float4
  static_assert((BN*BK) % (threads*4) == 0, ...);     // Bs 同理
  static_assert(BN % 4 == 0 && BM % 4 == 0, ...);     // float4 不跨行
  ```

- **autotune 子命令**:`./bench autotune M N K`,配置表(C++ 数组,放弃 X-macro 求可读)+ warmup + 3次取best。

### ★ 8×8 spilling 悬案闭环(v2 反例的续集)

- v2 记录:无 float4 时 8×8 触发 spill,Reg=128 触顶,GFLOPS 1358 腰斩。
- **Kernel 6 实测(float4 后)**:8×8(id=8)≈2538 GFLOPS,与 8×4 持平略低,**verify PASS,无 spill 崩盘**。
- **核心认知(portfolio 价值极高)**:**float4 把基础寄存器开销压低(96→64),给 acc 腾出空间,使之前必 spill 的 8×8 重新可行。优化之间存在依赖——一个优化解放了寄存器预算,另一个之前不可行的配置才变可行。** v2 的"寄存器预算天花板是单卡特性"补充为:**天花板也是单 kernel 实现特性,随前序优化移动。**

### ★ 扫描结果:甜点是平台,不是尖峰

隔离单测 + ncu Elapsed Cycles 裁决(墙钟噪声 >3% 不可信):

| 配置                  | Elapsed Cycles | Reg  | Occupancy | L1/TEX | 备注                       |
| --------------------- | -------------- | ---- | --------- | ------ | -------------------------- |
| 64×64×8 **8×4**(定版) | **83.4M**      | 64   | 98.8%     | 97.9%  | 🏆 cycle 最低、最省寄存器   |
| 64×64×**16** 8×4      | 83.7M          | 72   | 86.5%     | 98.3%  | 与 8×4 **cycle 仅差 0.3%** |
| 64×64×8 8×8           | ~等价          | —    | —         | —      | 不 spill,持平              |
| 128×128×8 8×8         | ~等价          | —    | —         | —      | 大 tile,持平               |

- **核心认知**:多个差异显著的配置(深 BK / 大 tile / 不同 TM·TN)**cycle 几乎相同**,因为它们都撞到**同一堵墙 L1/TEX ~98%**。BK=16 用更高寄存器(72)+更低 occupancy(86%)换到和 8×4 一样的 cycle——**殊途同归**。甜点是 plateau。
- **定版决策**:保持 **8×4**(cycle 最低、寄存器最省 64、occupancy 最高 98.8%,最稳)。BK=16 作等效备选记录。
- **float4 砍了搜索空间**:4×4、128×64 等配置因违反"每线程整数个 float4"约束被 static_assert 挡掉,编译失败。**向量化 vs 搜索灵活性的 trade-off**——siboehm 搬运处理余数所以能扫更大空间。

### ★ 方法论:WSL2 上如何做可信 benchmark

- 墙钟 GFLOPS 在本机抖动 >3%(同配置见过 2540~3054),来源:时钟 boost 行为 + 批量跑的缓存/热状态污染 + 配置在表中的位置。
- **批量 autotune 的绝对值与排序不可信**,只能粗筛。
- **Elapsed Cycles 是金标准**:测时钟周期数,与频率无关,不受 boost/节流干扰。精细排序必须用它。
- 隔离单测需:独立进程 + 之间 sleep 降温 + 各自 warmup + 取 best。

## 五、关键技术认知(已内化,本版新增标 ★)

1. ncu "Memory Throughput" 取内存子系统最忙一级,不等于 DRAM。
2. shared memory 和 L1 同一块物理 SRAM,ncu 的 L1/TEX 把 shared 访问也算进去。
3. 三级数据流 global → shared → register:shared 解决 block 级复用,register 解决 thread 级复用。
4. tile 大小(BM×BN)与线程排布(blockDim)解耦;2D tiling 让线程数大降而单线程活变多。
5. 行主序索引:`B[行][列]=行*N+列`(K×N 乘 N);`A[行][列]=行*K+列`(M×K 乘 K)。
6. 小尺寸 + 大 L2 掩盖访存瓶颈,性能锚点必须用大尺寸(256³ 仅 829 GFLOPS vs 4096³ 2308)。
7. ★ **L1/TEX throughput 受"访问指令数"驱动,不是数据字节量**。LSU 瓶颈是指令发射率。这是 float4 能卸压的根据。
8. ★ **throughput 是比率(流量÷时间),kernel 被 stall 拖慢时,所有 throughput 看着都"不忙"**。所以 spilling 不能靠 SOL 百分比判断,必须直接量 local memory 流量。
9. ★ **算术强度有寄存器预算天花板**:acc 留在寄存器是 2D tiling 收益成立的隐含前提,一旦 spill 到 local(DRAM),理论收益清零甚至倒亏。
10. ★ **float4 的真实收益常是寄存器红利,不是 L1 指令红利**:向量化让编译器寄存器分配更紧凑,可能间接拉满 occupancy。实测 L1/TEX 没降而 occupancy 62%→99%、性能因此提升(Kernel 5)。**优化的生效机制未必是预想的那个,必须 ncu 验。**
11. ★ **优化之间有依赖,寄存器天花板会移动**:前序优化(float4)解放寄存器后,之前必 spill 的配置(8×8)变可行。天花板不只是单卡特性,也是单实现特性。
12. ★ **性能甜点常是平台不是尖峰**:多个差异显著的配置可能 cycle 几乎相同,因为撞同一堵墙(L1/TEX 98%)。换 tile 尺寸在墙内挪动不破墙。
13. ★ **Elapsed Cycles 是抗噪金标准**:墙钟 GFLOPS 受时钟 boost/热节流/缓存状态干扰(WSL2 上 >3% 抖动),cycle 测周期数与频率无关,精细排序只信它。
14. ★ **向量化收窄 autotuning 搜索空间**:float4 要求每线程整数个 float4,部分在标量下合法的 tile 配置被 static_assert 挡掉。这是工程 trade-off,不是 bug。

## 六、待办 / 下一步

### 立即下一步:Kernel 7 —— Warp Tiling(siboehm Kernel 8)

- **动机**:Kernel 4/5/6 共同的硬墙是 **L1/TEX ~98%**——SRAM 子系统的访问指令发射率饱和。float4 没压下它,换 tile 尺寸也没用(autotuning 已证),唯一出路是**减少每线程对 shared 的访问次数**。
- **核心机制**:在 block tiling 和 thread tiling 之间插入 **warp tiling** 层。一个 warp 内的线程协作复用从 shared 取出的数据,把 shared 读放进寄存器在 warp 级复用,进一步摊薄 shared 访问指令数。
- **预期**:L1/TEX 终于从 98% 下降,Compute 上升。
- **难点(写时专攻)**:warp 内线程的二级坐标拆分(warp_id / lane_id),三套坐标真值表要再扩一层。延续"查表不靠记忆"纪律。
