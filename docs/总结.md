我觉得这十个 kernel 其实始终贯彻的是一个思想：围绕 GEMM 的数据复用和访存效率做 trade-off。

一开始 naive kernel 的问题很明显，A/B 都直接从 global memory 读，load 延迟很长，warp 经常卡在 long scoreboard 上。我们想优化 load 时间，所以引入 shared memory：先把 A/B 的 tile 从 global memory 搬到 shared memory，再让 block 内的线程反复复用这批数据。这里的重点不只是 shared memory 访问更快，而是把 global memory 的重复读取变成 shared memory 内部复用。

但是 shared memory 加入之后，我们又发现 MIO 大量拥堵。本质原因是 shared memory load/store 这些 MIO/LSU 指令太密集，尤其是每次 FMA 前都要从 shared memory 取 A/B，LDS 指令太多，把 MIO 队列堵住了。也就是说，你费了很大劲把数据搬进 shared memory，如果每个数只算一两次，复用不够，就很难把搬运成本赚回来。

所以后面引入 register tile。register tile 的核心是让一个 thread 负责多个 C 元素，把从 shared memory 读出来的 A/B 先放进寄存器里复用。这样一条 LDS 可以喂多个 FMA，降低 LDS 指令和 FMA 的比例，也能利用单线程内部的 ILP 来隐藏一部分延迟。

之后的 float4 读取/写回，主要是为了减少访存指令条数，提高单条指令的搬运宽度。但 float4 本身不是万能的，它要求地址连续并且对齐；如果线程到地址的映射设计不好，还是会出现 global memory 访存不合并，甚至 shared memory bank conflict。

这也是为什么后面要引入 warp tile。warp tile 不是单纯为了把更多数据从 shared memory 搬到寄存器，这件事主要是 register tile 在做。warp tile 更重要的是显式规定每个 warp 负责 C 的哪一块区域，以及 warp 内 32 个 lane 怎么排。这样可以让同一个 warp 的 global memory 访问更连续，写回 C 更容易合并，shared memory 访问也更容易利用 broadcast 或避免 bank conflict。

后面的 padding、转置 AS、调整 shared memory 布局这些技巧，主要都是为了消除 shared memory bank conflict，让 shared memory 的 load/store 不要被拆成太多 transaction。double buffer 则是为了把下一轮 global memory 到 shared memory 的加载和当前 tile 的计算重叠起来，隐藏加载延迟，而不是从硬件上真的消除延迟。

总的来说，这条优化路线不是简单地把数据搬到更快的存储空间，而是在 global memory、shared memory、register 之间不断做数据复用、访存合并、指令数量、bank conflict、occupancy 和 ILP 的 trade-off。shared memory 解决 global memory 重复读取的问题，register tile 解决 shared memory 读取指令太多的问题，warp tile 解决 warp 内线程协作和访存布局的问题，vectorized load/store、padding、double buffer 这些技巧则是在这个主线上进一步把读写效率做高。
